# Database Schema

This system uses SQLite with SQLAlchemy ORM models. The schema captures the agent hierarchy, policies, sales transactions, commission ledger entries, volume bonus accruals, and clawback workflows.

## Entity Relationships

- `agents` is a self-referencing table that stores hierarchy levels (Agent → Team Lead → Manager → Director).
- `policies` holds unique policy numbers and product metadata.
- `sales` ties a policy to the selling agent and records deal metadata, including a JSON snapshot of the selling hierarchy at the time of sale.
- `commission_ledger` logs every monetary entry generated by the engine (`FYC`, `OVERRIDE`, `VOLUME_BONUS`, `CLAWBACK`).
- `volume_bonus_accruals` keeps cumulative volume and bonus payouts per agent per period (monthly, quarterly, annual).
- `clawbacks` and `clawback_items` track cancellation workflows and the amounts recovered across agents and entry types.

The diagram below summarises core relationships:

```
agents ─┬──────────────┬──────── sales
        ├─< agents (parent/children)
        ├─< commission_ledger
        ├─< volume_bonus_accruals
        └─< clawback_items

policies ─┬─< sales
          ├─< commission_ledger
          └─< clawbacks

sales ─┬─< commission_ledger
       └─< clawback_items (via commission_ledger metadata)
```

## Table Definitions

### `agents`
- `id` INTEGER PK
- `name` VARCHAR(120) NOT NULL
- `email` VARCHAR(120) UNIQUE NOT NULL
- `password_hash` VARCHAR(256) NOT NULL
- `level` INTEGER NOT NULL (1=Agent, 2=Team Lead, 3=Manager, 4=Director)
- `parent_id` INTEGER FK → `agents.id` (nullable)
- `external_id` VARCHAR(32) (nullable)
- `active` BOOLEAN DEFAULT TRUE
- `avatar_url` VARCHAR(255) (nullable)

### `policies`
- `id` INTEGER PK
- `policy_number` VARCHAR(64) UNIQUE NOT NULL
- `product` VARCHAR(64) DEFAULT "Life"
- `fyc_rate` FLOAT DEFAULT 0.5

### `sales`
- `id` INTEGER PK
- `policy_id` INTEGER FK → `policies.id`
- `seller_id` INTEGER FK → `agents.id`
- `premium` FLOAT NOT NULL
- `sale_date` DATE NOT NULL (defaults to `date.today`)
- `customer_name` VARCHAR(120)
- `product` VARCHAR(120)
- `mobile` VARCHAR(30)
- `gender` VARCHAR(16)
- `notes` TEXT
- `hierarchy_snapshot` TEXT (stores JSON array of `{id, level, name}`)

### `commission_ledger`
- `id` INTEGER PK
- `agent_id` INTEGER FK → `agents.id`
- `sale_id` INTEGER FK → `sales.id` (nullable)
- `policy_id` INTEGER FK → `policies.id` (nullable)
- `entry_type` VARCHAR(32) NOT NULL (`FYC`, `OVERRIDE`, `VOLUME_BONUS`, `CLAWBACK`)
- `amount` FLOAT NOT NULL
- `date` DATE NOT NULL
- `meta` TEXT (JSON metadata, nullable)
- `created_at` DATETIME DEFAULT `utcnow()`

### `volume_bonus_accruals`
- `id` INTEGER PK
- `agent_id` INTEGER FK → `agents.id`
- `period_type` VARCHAR(12) NOT NULL (`monthly`, `quarterly`, `annual`)
- `period_start` DATE NOT NULL
- `period_end` DATE NOT NULL
- `total_volume` FLOAT NOT NULL DEFAULT 0.0
- `bonus_rate` FLOAT NOT NULL DEFAULT 0.0
- `bonus_paid` FLOAT NOT NULL DEFAULT 0.0
- `updated_at` DATETIME DEFAULT `utcnow()` with `onupdate`
- Unique constraint: (`agent_id`, `period_type`, `period_start`)

### `clawbacks`
- `id` INTEGER PK
- `policy_id` INTEGER FK → `policies.id`
- `cancellation_date` DATE NOT NULL
- `reason` VARCHAR(80) (nullable)
- `notes` TEXT (nullable)
- `status` VARCHAR(16) DEFAULT `PENDING` (`PENDING`, `APPROVED`, `DENIED`)
- `created_at` DATETIME DEFAULT `utcnow()`

### `clawback_items`
- `id` INTEGER PK
- `clawback_id` INTEGER FK → `clawbacks.id`
- `agent_id` INTEGER FK → `agents.id`
- `entry_type` VARCHAR(32) NOT NULL
- `original_amount` FLOAT NOT NULL
- `clawback_amount` FLOAT NOT NULL
- `meta` TEXT (JSON metadata, nullable)

## Sample Data

- `backend/commission.db` ships with representative agents, policies, sales, ledger entries, and bonus accruals so the UI and reports render immediately.
- To seed a fresh database with a minimal hierarchy, run:

  ```bash
  cd backend
  python -m seeds.seed
  ```

  This command creates a Director → Manager → Team Lead → two Agent structure with default credentials (`pass`) for quick authentication tests.

## Schema Evolution

- The backend calls `core.db.init_db()` on startup to create missing tables and add new columns if they do not exist (lightweight migration support when iterating locally).
- No destructive migrations run automatically; manual data migrations should be handled separately if the schema changes significantly.
